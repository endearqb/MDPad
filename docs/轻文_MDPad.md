# Windows 极轻量 Markdown 阅读/编辑器 轻文 MDPad（Tauri）需求文档（PRD）

> 目标：做一个 **Windows 可用、极轻量** 的 Markdown 阅读/编辑器，体验要像打开 `.txt` 一样顺滑。  
> 核心约束：**不要“先选目录/库/工作区”** 的产品形态；以 **单文件打开** 为中心。  
> 技术倾向：**Tauri v2 + React + Vite + TipTap**；UI 用 **BaseUI + TailwindCSS**；编辑交互倾向 **Novel 风格**，但 **暂时不需要 AI**，且 **不使用 Next.js**。

---

## 1. 背景与动机

用户希望在 Windows 上获得一种“像打开 txt 一样”的 Markdown 使用体验：

- 双击 `.md` 文件即可打开并呈现/编辑
- 不要求提前维护一个笔记目录或数据库
- 操作链路短：打开 → 阅读/编辑 → 保存

同时用户喜欢 `novel` 的 UI 和交互范式，希望编辑体验更接近 Notion/Novel，而不是传统 Markdown 编辑器。

---

## 2. 产品定位

**单文件 Markdown 阅读/编辑器（Desktop App）**  
- 以“打开一个文件”作为主入口，而不是“管理一个知识库/目录”
- 轻量、启动快、依赖少
- 支持阅读与编辑（编辑为必需）
- 不包含 AI 能力（暂时）

---

## 3. 目标与非目标

### 3.1 目标（Goals）
1. **像 txt 一样打开**：双击 `.md/.markdown` 直接打开应用并加载该文件
2. **不需要目录选择**：首次启动、日常使用都不要求选定笔记目录
3. **轻量启动快**：冷启动快、包体尽量小、资源占用低
4. **可编辑**：提供良好的富文本/块编辑体验（Novel 风格），并能保存回 Markdown
5. **多次打开体验好**：单实例模式，已打开时再双击文件 → 前台聚焦并切换到新文件
6. **基础文件工作流完整**：Open / Save / Save As，且支持拖拽打开文件

### 3.2 非目标（Non-goals）
- 不做 AI 自动补全/对话/写作（暂时不做）
- 不做知识库管理、目录扫描、全文索引、标签库、笔记同步（暂时不做）
- 不做多人协作、在线发布、云存储（暂时不做）
- 不以 Next.js/服务端渲染为基础（明确不采用）

---

## 4. 目标用户与使用场景

### 4.1 目标用户
- 经常查看/编辑 `.md` 的个人用户或开发者/独立创作者
- 需要随手打开一个 md 文件进行阅读或轻量编辑
- 不希望被“工作区/笔记库”束缚

### 4.2 核心场景
1. **资源管理器双击打开**：双击 `README.md` → 立刻打开并渲染/编辑
2. **拖拽打开**：把 `.md` 拖到窗口 → 打开该文件
3. **快捷键打开**：Ctrl+O → 选择一个 md 文件 → 打开
4. **编辑保存**：修改内容 → Ctrl+S 保存
5. **切换文件**：已运行时再次双击其他 md → 同一窗口切换到新文件

---

## 5. 体验原则（UX Principles）

1. **单文件中心**：界面和功能围绕“当前文件”组织  
2. **少步骤**：打开/保存动作 1～2 步完成  
3. **无库/无目录强制**：不出现“请选择笔记目录”的阻断流程  
4. **默认轻量**：默认不开重型功能（索引/同步/AI 等）  
5. **可预期**：保存行为清晰、路径明确；dirty 状态可见

---

## 6. 功能需求（Functional Requirements）

### 6.1 文件打开方式（Must）
- **文件关联打开（Windows）**
  - 安装后注册 `.md` / `.markdown` 文件关联
  - 双击 md 文件启动 app，并自动打开该文件
- **单实例模式**
  - 若应用已运行，再次通过文件关联打开另一个 md：
    - 不新开窗口（默认）
    - 现有窗口置前、聚焦
    - 切换到新文件并加载
- **拖拽打开**
  - 将 md 文件拖入窗口：打开并加载
- **菜单/快捷键打开**
  - Ctrl+O：打开文件对话框，仅筛选 `.md/.markdown`

#### 6.1.1 打开行为规则
- 打开新文件时，如当前文件 **dirty**（未保存修改）：
  - 弹窗提示：保存 / 不保存 / 取消
  - 默认建议：保存（但不强制）

---

### 6.2 编辑与渲染（Must）
- **编辑器形态**：Novel 风格块编辑（基于 TipTap）
- **Markdown 兼容**
  - 能导入 `.md` 内容 → 编辑器内正确呈现（至少支持常见语法）
  - 能从编辑器导出 Markdown → 保存回文件
- **基础格式能力（第一期必须）**
  - 标题（H1-H3）
  - 粗体/斜体/删除线
  - 无序/有序列表
  - 引用
  - 代码块（至少纯文本；语法高亮可选）
  - 链接
  - 分割线
- **Slash Command（期望）**
  - `/` 触发命令菜单，快速插入块类型（可用 novel 现成交互实现）

> 备注：图片、表格、任务列表、callout 等可作为后续迭代。

---

### 6.3 保存与文件管理（Must）
- **Save（Ctrl+S）**
  - 若有 currentPath：直接写回该文件
  - 若无 currentPath：触发 Save As
- **Save As（Ctrl+Shift+S）**
  - 选择保存路径与文件名，默认扩展名 `.md`
- **dirty 状态**
  - 内容变更后显示未保存状态（标题栏 `*` 或 UI 指示）
- **崩溃/关闭保护**
  - 关闭窗口时若 dirty：提示保存

---

### 6.4 UI 需求（Must）
- **简洁顶部栏**（BaseUI）
  - 文件名（路径可折叠显示）
  - 保存状态（已保存/未保存）
  - 常用按钮：Open / Save（可选显示）
- **编辑区域**（Tailwind typography + editor style）
  - 宽度舒适（如类似 Notion 的正文宽度）
  - 支持浅色/深色（至少跟随系统或提供切换）
- **不出现目录选择/库管理 UI**
  - 不出现“选择 notes folder”
  - 不出现“建立工作区/数据库”向导

---

## 7. 非功能需求（Non-functional Requirements）

### 7.1 性能
- 冷启动目标：可感知的“秒开”体验（尽量减少重依赖）
- 打开 1–5MB Markdown：可用，不卡死（可接受轻微延迟）
- 内存占用：尽量低（避免常驻索引服务/后台任务）

### 7.2 可靠性
- 写入失败要有明确错误提示与回退策略（例如提示另存为）
- 文件路径包含中文、空格、特殊字符要稳定支持

### 7.3 可维护性
- 前后端解耦：文件 IO / 单实例 / 系统对话框交给 Tauri 插件层
- 编辑器能力模块化：扩展可开关（AI 未来可插拔）

---

## 8. 技术方案倾向（Implementation Preferences）

### 8.1 技术栈
- Desktop Shell：**Tauri v2**
- Frontend：**React + Vite**
- Editor：**TipTap**
- UI：**BaseUI + TailwindCSS**
- Markdown 解析/导出：
  - 导入：Markdown → TipTap 文档结构
  - 导出：TipTap → Markdown

### 8.2 插件/能力
- 打开/保存对话框：Tauri dialog plugin
- 文件读写：Tauri fs plugin
- 单实例：Tauri single-instance plugin
- 文件关联：Tauri bundle.fileAssociations 配置（Windows）

> 注意：文件关联打开时的 argv 传递、以及已运行实例的 argv 转发，是“txt 体验”的关键闭环。

## 9. 里程碑（Milestones）

### M0：可运行骨架（1–2 天）
- Tauri + React/Vite 启动
- BaseUI + Tailwind 初始化
- 空白页面 + 顶部栏

### M1：文件工作流闭环（2–4 天）
- Ctrl+O 打开 md，读取并显示
- Ctrl+S 保存 / Ctrl+Shift+S 另存为
- dirty 状态 & 关闭保护弹窗

### M2：编辑器引入（3–6 天）
- TipTap 基础编辑能力
- Markdown 导入/导出闭环
- 基础格式（标题/列表/引用/代码块/链接）

### M3：txt 体验完成（2–4 天）
- Windows 文件关联（双击打开）
- 单实例 argv 转发（已运行时切换文件）
- 拖拽打开文件

### M4：体验打磨（持续）
- 主题（浅色/深色）
- Slash command 菜单
- 更好的排版与快捷键
- 错误处理与提示完善

## 10. 风险与对策

1. **TipTap Markdown 导入/导出一致性**
   - 风险：编辑器结构与 Markdown 互转丢信息或格式走样
   - 对策：第一期只承诺“常见语法子集”，复杂语法后续增强

2. **文件关联 + 单实例在 Windows 的边界行为**
   - 风险：argv 传递、路径带空格/中文、二次启动行为不一致
   - 对策：严格测试：路径编码、空格、长路径、不同打开方式（双击/右键打开方式/拖拽）

3. **“轻量”目标被需求膨胀侵蚀**
   - 风险：加入索引/库管理/同步导致变重
   - 对策：明确一期范围：单文件 + 基础编辑 + 文件关联；其他全放后续

## 11. 验收标准（Acceptance Criteria）

### A. 像 txt 一样打开（必须全部通过）
- [ ] 安装后，双击 `.md` 文件可直接打开该文件
- [ ] 应用已运行时，双击另一个 `.md` 不会新开窗口，而是聚焦并切换到新文件
- [ ] 拖拽 `.md` 到窗口能打开
- [ ] Ctrl+O 可打开 md，Ctrl+S 可保存，Ctrl+Shift+S 可另存为

### B. 不出现“目录选择/库管理”
- [ ] 首次启动不弹“选择笔记目录/工作区”
- [ ] 全流程无“导入目录/管理库”的强制步骤

### C. 编辑体验（一期）
- [ ] 可编辑：标题/列表/引用/代码块/链接
- [ ] Markdown 导入后显示合理
- [ ] 保存后文件内容为 Markdown，可被其他编辑器打开
