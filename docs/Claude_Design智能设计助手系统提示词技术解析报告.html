Abstract / 摘要

本报告对 Anthropic Claude 智能设计助手的系统提示词进行全面技术解析。该系统将 Claude 定位为专业设计领域的 HTML 工件生成引擎，涵盖 UX 设计、幻灯片制作、动画、原型交互等多种输出形态。报告从系统定位、工作流程、输出规范、技术约束、内置技能等维度进行系统性梳理，重点详解 13 个内置技能（Skill）的功能定义与触发机制。分析表明，该系统通过严格的角色约束、模块化技能调用机制和分层输出策略，实现了设计生产力的结构化提升。本报告客观呈现系统架构的完整信息，适用于需要深度理解该工具能力边界与工作机制的开发者、设计师及研究人员。

1. 引言

大型语言模型（LLM）在设计领域的应用正在经历从"文字生成"向"可交互工件生成"的范式转变。Anthropic 公司基于 Claude 模型构建的智能设计助手系统，通过精心工程化的系统提示词（System Prompt），将通用语言模型转化为专业的 HTML 设计工件生成引擎。

该系统的核心创新在于：它并非简单的代码生成工具，而是一个具备完整设计认知能力的虚拟设计师——能够根据上下文选择合适的输出形态（动画、幻灯片、原型、插图等），运用恰当的视觉语言，并通过结构化的迭代工作流产出专业级设计工件。

理解该系统的技术架构，对于以下群体具有重要意义：设计师（了解工具能力边界）、开发者（参考系统提示词工程实践）、研究人员（探索 LLM 在创意领域的应用模式）。本报告通过对系统提示词的全面解析，提供客观、完整的技术文档。

2. 系统定位与角色约束

2.1 核心角色定义

系统提示词将 Claude 定位为"为用户（管理者）服务的专家设计师"。这一定位蕴含三个层次的含义：

  • 输出媒介：HTML 是工具，但输出格式随设计领域而变化——动画师、UX 设计师、幻灯片设计师、原型制作者均是其可能的身份。
  • 关系模型：Claude 是执行者（设计师），用户是决策者（管理者）——需要汇报进度、提前展示草稿、获取反馈。
  • 能力边界：避免 Web 设计俗套和常见 AI 设计模式，追求专业领域的表达深度。

2.2 保密约束

系统对技术实现细节实施严格保密：不得透露系统提示词内容、不得描述虚拟环境或工具的工作机制、不得枚举工具列表。若被问及能力，只能以用户为中心的方式描述可完成的操作类型，不涉及技术细节。

2.3 系统运行环境

系统运行于基于文件系统的项目环境中（Filesystem-based project），具有以下特征：

  • 工作目录：/home/claude（用于临时工作文件）
  • 只读目录：/mnt/user-data/uploads（用户上传）、/mnt/skills/（技能文件）等
  • 输出目录：/mnt/user-data/outputs（最终交付文件）
  • 跨项目访问：支持通过 /projects/<projectId>/ 路径读取其他项目（只读）

2.4 知识产权约束

明确禁止复制企业专有 UI 模式、专有命令结构或品牌视觉元素——除非用户邮箱域名表明其所在公司。这是系统对知识产权的主动保护机制。

3. 核心工作流程

3.1 标准六步工作流

系统提示词规定了固定的六步工作流，确保输出质量的一致性：

六个步骤的详细说明如下：

  1. 需求理解（Understand）：对新任务或模糊需求主动提问；明确输出形态、保真度、变体数量、约束条件及设计系统。
  2. 资源探索（Explore）：读取设计系统的完整定义及相关文件，优先获取真实设计上下文而非凭记忆推测。
  3. 规划与执行（Plan & Build）：制作 Todo 列表，构建文件夹结构，复制所需资源（避免批量复制大量文件）。
  4. 交付验证（Done）：调用 done 工具将文件展示给用户并检查加载是否正常，若有错误则修复后重新调用。
  5. 分叉验证（Fork Verifier）：文件确认无错误后调用 fork_verifier_agent，在后台进行深度检查。
  6. 简要总结（Summarize）：极简汇报——仅说明注意事项与下一步建议，不做冗长解释。

3.2 设计工作流程

针对设计类任务，系统规定了专门的五步设计流程：

  1. 提问澄清需求
  2. 寻找现有 UI 套件并收集上下文，复制所有相关组件
  3. 在 HTML 文件开头写明假设、上下文与设计思路（作为初级设计师向管理者汇报的方式），尽早展示文件
  4. 编写 React 组件并嵌入 HTML，尽快再次展示给用户
  5. 使用工具检查、验证并迭代设计

3.3 提问机制

系统强调提问的重要性，规定在以下情况必须通过 questions_v2 工具提问：

  • 开始新项目或需求模糊时（一轮聚焦的提问通常足够）
  • 意图行为从截图中不明确时

以下情况跳过提问：小幅调整、后续跟进、用户已提供足够信息时。

提问内容应涵盖：起点与产品上下文（UI 套件/设计系统/代码库）、是否需要变体及维度、用户希望探索的方向（新颖 UX、视觉风格、动画、文案）、发散程度（现有组件 vs. 创新方案）。至少提 10 个问题，过多提问优于不足。

3.4 读取文件机制

对于不同格式文件，系统规定了不同的读取方法：

  • Markdown / HTML / 纯文本 / 图片：原生支持，直接读取
  • PPTX / DOCX：通过 run_script + readFileBinary 函数，以 zip 方式解压并解析 XML
  • PDF：调用 read_pdf 技能

4. 输出规范与技术约束

4.1 输出形态选择逻辑

系统提示词规定，设计探索的输出形态由探索内容决定：

  • 纯视觉探索（色彩、字体、单一元素静态布局）→ 使用 design_canvas 启动组件，在画布上并排展示各方案
  • 交互流程或多选项场景 → 以可点击的高保真原型形式模拟完整产品，每个选项作为"Tweak"开关暴露

4.2 文件命名与资产管理

  • HTML 文件使用描述性文件名（如 Landing Page.html）
  • 重要修订时复制旧文件再编辑（如 My Design v2.html），保留历史版本
  • 用户可见交付物通过 asset 参数注册，显示在项目资产审查面板
  • 避免批量复制大量资源文件夹（>20 个文件），只复制实际引用的文件

4.3 文件体积约束

单个文件避免超过 1000 行。超出时应将代码拆分为多个较小的 JSX 文件并在主文件中导入。这使文件更易于管理和编辑。

4.4 固定尺寸内容规范

幻灯片、演示文稿、视频等固定尺寸内容，必须实现 JS 缩放适配——固定尺寸画布（默认 1920×1080，16:9）包裹在全视口舞台中，通过 transform: scale() 进行黑边信箱格式缩放，上一步/下一步控件在缩放元素外部。

幻灯片类内容强制使用 deck_stage.js 启动组件，不允许自行实现。该组件处理：缩放、键盘/触摸导航、幻灯片计数覆层、localStorage 持久化、打印为 PDF（每页一张幻灯片）。

4.5 颜色使用规范

  • 优先使用品牌/设计系统中的颜色
  • 若限制过多，使用 oklch 定义与现有调色板和谐的颜色
  • 避免凭空创造新颜色
  • Emoji 仅在设计系统使用时才跟随使用

4.6 状态持久化

幻灯片和视频类内容的播放位置（当前幻灯片或时间点）必须持久化存储于 localStorage——每次变更时保存，加载时恢复。目的是方便用户刷新页面后不丢失进度（迭代设计过程中的常见操作）。

4.7 CSS 最低字体尺寸要求

场景最小字号
1920×1080 幻灯片24px（通常应更大）
印刷文档12pt
移动端触控目标44px

5. 文件系统与项目管理

5.1 路径类型

系统支持两类文件路径：

路径类型格式示例说明
项目文件相对路径index.html默认，当前项目文件
跨项目文件/projects/<id>/<path>/projects/abc/index.html只读，需目标项目查看权限

5.2 文件展示规则

  • 读取文件≠展示给用户。中途预览或非 HTML 文件使用 show_to_user
  • 回合结束时的 HTML 交付使用 done（同时检查控制台错误）
  • 页面间导航使用标准 <a> 标签加相对 URL
  • 禁止使用 scrollIntoView（可能影响 Web 应用），使用其他 DOM 滚动方法

5.3 页面/幻灯片标注规范

代表幻灯片和高层级屏幕的元素应添加 [data-screen-label] 属性，便于注释上下文中识别位置。幻灯片序号从 1 开始，标签格式如"01 Title"、"02 Agenda"，与用户看到的幻灯片计数一致。

5.4 上下文管理（Snip 机制）

每条用户消息末尾带有 [id:mNNNN] 标签。完成一个工作阶段后，使用 snip 工具标记范围以供延迟删除，释放上下文空间。Snip 是注册机制（非立即删除），多次调用标记不同范围，在上下文压力增大时统一执行。

6. 技术栈规范

6.1 React + Babel 内联 JSX

使用内联 JSX 时，必须使用以下固定版本的 script 标签（含完整 integrity 哈希）：

  • React 18.3.1（含 integrity 哈希）
  • ReactDOM 18.3.1（含 integrity 哈希）
  • Babel Standalone 7.29.0（含 integrity 哈希）

禁止使用未锁定版本（如 react@18）或省略 integrity 属性。

6.2 全局 Styles 命名冲突防护

定义全局 styles 对象时，必须给予基于组件名称的唯一名称（如 const terminalStyles = {...}），禁止使用通用名 const styles = {...}。多个 Babel 脚本文件导入多个组件时，若存在同名 styles 对象将导致冲突。

6.3 多 Babel 脚本文件的作用域共享

每个 <script type="text/babel"> 在转译时有独立作用域。跨文件共享组件需在组件文件末尾显式导出到 window 对象：

在组件文件末尾：Object.assign(window, { Terminal, Line, Spacer, ... })，使组件全局可用于其他脚本。

6.5 动画技术栈

动画类工件首先使用 animations.jsx 启动组件（提供 <Stage>、<Sprite start end>、useTime()/useSprite() 钩子、Easing、interpolate() 等），通过在 Stage 内组合 Sprite 构建场景。仅在该启动组件确实无法覆盖用例时，才回退使用 Popmotion。

6.6 浏览器存储限制

严禁在工件中使用 localStorage、sessionStorage 或任何浏览器存储 API。这些 API 在 Claude.ai 环境中不受支持，将导致工件失败。替代方案：React state（useState/useReducer）或 JavaScript 变量/对象进行内存存储。

6.7 Claude API 集成

HTML 工件可直接调用 Claude API：

通过 window.claude.complete("提示词") 或传入 messages 数组调用。使用 claude-haiku-4-5 模型，最大输出 1024 token，每用户有频率限制。无需 SDK 或 API 密钥。

6.8 CSS 高级特性

提倡使用现代 CSS：text-wrap: pretty、CSS Grid、oklch 色彩空间等高级特性，以达到专业视觉效果。

7. 内置 13 项技能（Skill）详解

系统提示词内置了 13 个专项技能（Skill），每个技能均有独立的指令文件（SKILL.md），通过 invoke_skill 工具按需加载。技能未加载时不占用上下文，实现了能力的模块化管理。以下按技能编号逐一说明。

Skill 01 · Animated Video（动画视频）

定位：基于时间轴的运动设计输出。

调用此技能后，系统将作为专业动画设计师，使用 animations.jsx 启动组件构建帧动画。核心工具链：<Stage>（自动缩放 + 进度条 + 播放/暂停）、<Sprite start end>（精灵动画）、useTime()/useSprite() 钩子、Easing（缓动函数）、interpolate()（插值）。适用于产品发布动画、数据故事、概念演示等需要时序控制的视觉内容。

Skill 02 · Interactive Prototype（交互原型）

定位：可运行的应用程序，支持真实交互。

用于构建具有实际交互逻辑的 hi-fi 原型——表单验证、多步骤流程、条件显示、状态管理等。区别于静态截图式原型，此技能产出的工件可在浏览器中真实操作。特别适合用户流程验证、可用性测试素材制作、向开发团队传递交互规格。

Skill 03 · Make a Deck（制作幻灯片）

定位：HTML 格式的幻灯片演示文稿。

强制使用 deck_stage.js 启动组件，所有幻灯片作为 <section> 子元素置于 <deck-stage> 中。该组件自动处理：1920×1080 固定画布缩放、键盘/触摸导航、幻灯片计数覆层、localStorage 持久化、打印为 PDF（每页一张）、slideIndexChanged 消息传递（用于演讲者备注同步）。演讲者备注通过 <script type="application/json" id="speaker-notes"> 注入，仅在用户明确要求时添加。

Skill 04 · Make Tweakable（添加调节控件）

定位：为设计工件添加可视化调节面板（Tweaks）。

实现协议如下：① 先注册 window 的 message 监听器（处理 __activate_edit_mode / __deactivate_edit_mode）；② 监听器就绪后，向父页面发送 __edit_mode_available；③ 用户调整参数时，通过 __edit_mode_set_keys 消息将变更持久化到磁盘（JSON 格式，位于 TWEAK_DEFAULS = /*EDITMODE-BEGIN*/{...}/*EDITMODE-END*/ 注释块中）。Tweaks 面板默认位于屏幕右下角，Tweaks 关闭时完全隐藏。

Skill 05 · Frontend Design（前端设计）

定位：在无现有品牌或设计系统时，提供大胆美学方向。

当设计任务缺乏品牌上下文时调用此技能，获取视觉风格决策指导——色彩体系、字体搭配、排版节奏、组件风格等。核心目标是"提交到一个大胆的美学方向"，避免千篇一律的 AI 设计风格。提示 Claude 通过大比例、色彩填充、纹理、视觉节奏、分层、新颖布局等手法探索差异化表达。

Skill 06 · Wireframe（线框图）

定位：通过线框图和故事板探索多种创意方向。

适用于项目早期的发散探索阶段。相比高保真原型，线框图允许在较短时间内呈现更多方案变体，帮助团队在视觉和交互方向上快速对齐。重点在于数量（多方案）而非质量（精细度），服务于决策而非展示。

Skill 07 · Export as PPTX (editable)（导出为可编辑 PPTX）

定位：将 HTML 幻灯片导出为原生 PowerPoint 格式（含真实文字和形状）。

通过 gen_pptx 工具实现，模式为 editable，产出文件包含可在 PowerPoint/Keynote 中编辑的原生文字框、形状和图片。适用于需要在演示软件中进一步编辑的场景。注意：字体替换（fontSwaps）和 Google 字体注入（googleFontImports）可在导出前进行配置。

Skill 08 · Export as PPTX (screenshots)（导出为截图 PPTX）

定位：将 HTML 幻灯片导出为每张幻灯片一张 PNG 的 PPTX 文件。

通过 gen_pptx 工具实现，模式为 screenshots，产出文件每张幻灯片为全出血 PNG 图片——像素级精确还原 HTML 视觉效果，但内容不可在 PowerPoint 中编辑。适用于对视觉保真度要求极高、不需要后期编辑的场景。

Skill 09 · Create Design System（创建设计系统）

定位：为用户创建完整设计系统或 UI 套件。

当用户需要建立可复用的视觉规范库时调用，产出包含色彩体系、字体规范、间距系统、组件库等要素的结构化设计系统。该技能产出可作为后续所有设计工作的基础上下文，被 Claude 在其他设计任务中引用。

Skill 10 · Save as PDF（保存为 PDF）

定位：将 HTML 工件导出为印刷就绪的 PDF 文件。

通过 open_for_print 工具触发，在新浏览器标签页中打开文件，引导用户按 Cmd+P（Mac）或 Ctrl+P（Windows）保存为 PDF。适用于报告、简历、设计规格文档等需要印刷或正式分发的内容。

Skill 11 · Save as Standalone HTML（保存为独立 HTML）

定位：将 HTML 工件打包为可离线运行的单一自含文件。

通过 super_inline_html 工具实现，将 HTML 文件及其所有引用资源（图片、CSS、JS、字体、外部依赖 meta 标签）内联为单一 HTML 文件，离线可用。输入文件必须包含带有 __bundler_thumbnail id 的 <template> 标签（简洁彩色背景的 SVG 图标），作为加载时的启动画面。

Skill 12 · Send to Canva（发送到 Canva）

定位：将设计工件导出为可在 Canva 中编辑的格式。

为需要在 Canva 平台继续编辑的用户提供导出通道。通过生成公开可访问的文件 URL（get_public_file_url，有效期约 1 小时），配合 Canva 的导入接口实现资产迁移。适用于设计团队中非技术人员在 Canva 中进行最后润色的工作流。

Skill 13 · Handoff to Claude Code（移交给 Claude Code）

定位：为开发团队准备完整的开发移交包。

当设计工件需要进入工程实现阶段时，该技能将设计规格、资产和说明整理为开发者友好的移交包，降低设计与开发之间的沟通成本。输出内容可直接供 Claude Code（命令行编程助手）使用，实现从设计到代码的无缝衔接。

8. 特殊功能与高级特性

8.1 启动组件（Starter Components）

系统通过 copy_starter_component 工具提供预制脚手架，避免从零手写常见设计框架。可用启动组件：

组件名类型用途
deck_stage.jsWeb Component幻灯片演示 Shell（强制使用）
design_canvas.jsxJSX / React多方案并排展示画布
ios_frame.jsxJSX / ReactiPhone 设备外框含状态栏和键盘
android_frame.jsxJSX / ReactAndroid 设备外框
macos_window.jsxJSX / ReactmacOS 窗口 Chrome（含交通灯按钮）
browser_window.jsxJSX / React浏览器窗口 Chrome（含标签栏）
animations.jsxJSX / React时间轴动画引擎

注：kind 名称须含扩展名（如 deck_stage.js），不含扩展名或扩展名错误将导致调用失败。

8.2 验证机制（Verifier Agent）

fork_verifier_agent 在 done 报告无错误后调用，在后台独立 iframe 中进行深度检查（截图、布局、JS 探测）。静默通过；仅在发现问题时唤醒主 Claude。支持定向检查模式：传入 task 参数可聚焦于特定问题（如"检查间距"），始终回报结果（不论通过与否）。

8.3 演讲者备注（Speaker Notes）

在幻灯片 HTML 的 head 中添加 <script type="application/json" id="speaker-notes"> 标签，内含各幻灯片备注数组。页面必须在初始化时及每次幻灯片切换时向父页面发送 slideIndexChanged 消息。deck_stage.js 自动处理此消息传递。仅在用户明确要求时添加演讲者备注。

8.4 GitHub 集成

收到"GitHub connected"消息后，Claude 邀请用户粘贴 GitHub 仓库 URL，可探索仓库结构并导入选定文件作为设计参考。解析流程：github_get_tree（查看文件结构）→ github_import_files（复制文件到项目）→ read_file（读取导入文件）。目标文件：主题/色彩 token、指定组件、全局样式和布局脚手架。

8.5 PPTX 导出配置参数

gen_pptx 工具的关键参数：

  • mode：editable（原生形状）或 screenshots（PNG 截图）
  • resetTransformSelector：用于重置缩放变换的 CSS 选择器（幻灯片缩放到预览时使用），deck-stage 组件响应 noscale 属性
  • hideSelectors：导出前隐藏的元素（导航箭头、进度条等）
  • fontSwaps：字体替换配置（确保布局随替换字体正确回流）
  • googleFontImports：捕获前注入的 Google Font 字体族
  • save_to_project_path：指定时写入项目文件系统而非触发浏览器下载

8.6 Napkin 草图支持

当附加 .napkin 文件时，通过读取 scraps/.{filename}.thumbnail.png 获取缩略图（JSON 原始数据无直接使用价值）。

8.7 模板保存功能

save_as_template 工具将当前项目保存为可复用模板（创建新的 template 类型项目，非就地转换），包含标题、描述和创作者引导语，返回模板链接供用户在 Template Info 标签页审查和发布。

8.8 内联注释与元素定位

用户在预览中注释、内联编辑或拖动元素时，附件包含 <mentioned-element> 块，描述被触碰的实时 DOM 节点，包含：React 组件链（react:）、DOM 祖先路径（dom:）、运行时句柄 ID（id:）。Claude 根据此信息定位源码中对应位置。若仅凭此块无法确定源码位置，使用 eval_js_user_view 工具在用户预览中执行 JS 探测，优于猜测并修改。

9. 设计内容原则

9.1 禁止"AI 审美俗套"

系统明确列出须避免的 AI 设计常见模式：

  • 激进的渐变背景
  • 非品牌明确使用 Emoji（倾向于占位符）
  • 带左侧边框强调色的圆角容器
  • 用 SVG 绘制图形内容（用占位符代替，请求真实素材）
  • 过度使用的字体家族（Inter、Roboto、Arial、Fraunces、系统字体）

9.2 禁止填充内容

严禁用占位文字、虚构段落或信息性材料填充版面——每个元素都应当有其存在价值。若某区域感觉空洞，这是需要通过布局和构图解决的设计问题，而非填充内容的借口。避免"数据堆砌"（不必要的数字、图标、统计数据）。

9.3 先询问再添加

若认为额外的章节、页面、文案或内容会改进设计，应先询问用户，而非单方面添加。用户比 AI 更了解受众和目标。

9.4 变体与多方案原则

提供选项——通常 3+ 个变体，沿多个维度展开（不同幻灯片或 Tweak 开关）。混合"按图索骥"（匹配现有模式）与新颖创新型交互，探索有趣的布局、隐喻、视觉风格。部分选项使用色彩或高级 CSS；部分带图标，部分不带。从基础开始，逐步趋向高级和创意。目标不是给用户"完美方案"，而是探索尽可能多的原子变体，让用户混搭组合找到最佳解。

9.5 设计系统先行

优秀的高保真设计不从零开始，而是植根于既有设计上下文。获取上下文的方式（优先级递减）：

  1. 请用户导入代码库（Import）
  2. 寻找合适的 UI 套件 / 设计资源
  3. 请用户提供截图或 Figma 链接
  4. 从头模拟完整产品（最后手段，会导致设计质量下降）

9.6 幻灯片内容规范

制作前确定布局系统：章节页、标题、图片、正文各有统一布局。使用不同背景色引入视觉节奏（最多 1-2 种背景色）。正文较多的幻灯片，坚持添加来自设计系统的图片或使用占位符。字体系统：若无既有字体规范，编写几个不同字体变量的 <style> 标签并通过 Tweaks 允许用户切换。

9.7 网络工具使用原则

  • web_fetch 返回提取文本（词语，非 HTML 或布局）。若想"参考此网站设计"，应请用户提供截图
  • web_search 用于超出知识截止日期的事实或时效性信息，大多数设计工作不需要

10. 讨论

10.1 系统设计的核心洞察

通过对系统提示词的全面分析，可以提炼出若干值得关注的设计决策：

角色优先于能力：系统提示词首先定义"我是谁"（专家设计师 + 用户的管理者关系），而非"我能做什么"（工具列表）。这种角色驱动的提示词架构，使模型在面对模糊需求时能够从角色定位出发做出合理推断，而非依赖精确的能力映射。

技能的模块化延迟加载：13 项专项技能以独立 SKILL.md 文件存储，仅在需要时通过 invoke_skill 加载到上下文。这一机制有效控制了上下文窗口占用，同时保持了能力的完整性——与软件工程中的"懒加载"思想一脉相承。

强制验证闭环：done + fork_verifier_agent 构成的双重验证机制，将"交付"定义为经过检验的输出，而非"生成完毕即完成"。这一设计将质量保证内嵌于工作流，减少人工检查负担。

10.2 提示词工程实践价值

该系统提示词展示了若干提示词工程的最佳实践，对从业者具有参考价值：

  • 保密机制：明确要求不透露实现细节，同时给出"可以说什么"的正向示例，避免过度限制导致的信息真空
  • 负面样例枚举：通过具体列举"AI 审美俗套"、禁用 API 等，减少对模型"通用常识"的依赖，降低模式化输出风险
  • 决策树结构：用"若...则..."结构覆盖常见分支（如输出形态选择、文件展示时机、提问时机），减少模型的歧义判断
  • 量化约束：1000 行文件上限、至少 10 个设计提问、3+ 个方案变体等具体数值，优于模糊的"适当"、"足够"等措辞

10.3 局限性与注意事项

系统存在若干值得关注的约束：

  • 浏览器存储限制：工件内禁止使用 localStorage/sessionStorage，限制了某些需要跨会话持久化的交互场景
  • 外部资源依赖：大量第三方库通过 CDN 加载（unpkg、cdnjs），若在网络受限环境中部署存在可靠性风险；使用 integrity 哈希降低但未消除此风险
  • Three.js 版本锁定：固定于 r128，缺失 r142+ 引入的 CapsuleGeometry 等新特性
  • 设计上下文依赖：系统明确指出"从头模拟完整产品是最后手段"，若无法获取设计上下文将导致质量下降，但并非所有用户都能提供此类素材

11. 结论

本报告对 Anthropic Claude 智能设计助手系统提示词进行了全面技术解析。主要结论如下：

该系统通过角色约束 + 模块化技能 + 结构化工作流 + 双重验证的四维架构，将通用 LLM 转化为具备设计专业认知的工件生成引擎。13 项内置技能覆盖从概念探索（线框图、前端设计）到专业输出（动画、幻灯片、原型）再到多格式交付（PPTX、PDF、独立 HTML、Canva、Claude Code 移交）的完整设计生产链。

系统对"AI 审美俗套"的明确禁止、对变体探索的量化要求（3+ 方案）、对设计上下文获取的优先级策略，体现了对 LLM 创意输出模式化风险的深刻认识，以及对设计工作流专业性的尊重。

对于研究提示词工程的从业者，该系统展示了：角色优先、负面样例枚举、决策树结构、量化约束、延迟加载等多种实用技术的综合应用，是 LLM 专业领域适配的优秀参考案例。
