# SVG 编辑功能完整重写需求文档（2026-04-21 15:00）

## 背景与问题
- 当前用户反馈的直接问题是“SVG 的编辑弹窗没有弹出”。这不是单个按钮或单个 modal 的局部故障，而是整条 SVG 编辑链路已经变成高耦合、脆弱、难以维护的状态。
- 现有 SVG 编辑横跨多层：
  - iframe 预览宿主脚本中的命中检测、右键菜单、双击/按下触发。
  - 父层 `HtmlPreview` 的消息消费、状态管理、modal 打开与 HTML 回写。
  - `htmlPreviewEdit` 中的 SVG 请求/patch/source snapshot 类型与写回逻辑。
  - `SvgTextCanvasEditor` 及其配套几何、能力、session、样式与测试。
- 这条链路的已知脆弱点包括：
  - 触发依赖 iframe 内事件命中与消息桥，真实 WebView 环境下容易出现“看得到元素，但编辑请求没稳定到父层”的问题。
  - 宿主脚本、父层状态和 modal 编辑器之间耦合过深，入口、选择、frame、draft、commit、sync 分散在多份代码中。
  - SVG 编辑代码量和状态面明显大于 chart 编辑，但两者又共处同一套 HTML preview 桥接体系，导致局部修复很容易误伤其他行为。
  - 现有实现已经有多轮“修入口/修消息顺序/修 modal 触发”的历史，说明问题不是单点 bug，而是架构已经不适合继续修补。

## 现有功能盘点

### 入口与触发
- inline SVG 左键命中后可尝试直接打开 SVG 编辑器。
- inline SVG 双击时可尝试直接打开 SVG 编辑器。
- inline SVG 右键菜单支持 `Edit SVG`。
- 父层 `HtmlPreview` 还能直接消费 iframe 发来的 `open-svg-editor` / `svg-selection` 等请求后打开 modal。

### 当前可编辑对象
- 基础图形：`rect`、`circle`、`ellipse`、`line`。
- 点集图形：`polygon`、`polyline`。
- 路径图形：`path`。
- 文字图形：`text`、`tspan`。
- 运行时还会尝试区分 `shape` / `connector` / `text` 三类对象。

### 当前支持的编辑能力
- 几何编辑：
  - `rect` 的 `x/y/width/height/rx/ry`
  - `circle` / `ellipse` 的圆心与半径
  - `line` 的两端点
  - `polygon` / `polyline` 的 `points`
  - `path` 的 `d`
  - `text` / `tspan` 的 `x/y`
- 样式编辑：
  - `fill`、`stroke`、`stroke-width`、`opacity`
  - 文字相关：`font-size`、`text-anchor`、`font-family`
  - 连线相关：`marker-start`、`marker-end`、`stroke-dasharray`、`stroke-linecap`、`stroke-linejoin`
- 画布交互：
  - 拖动元素
  - 多选
  - connector 命中扩展与正交路由相关处理
  - path / polyline / polygon 点级编辑
  - Undo / Redo
- 校验与保护：
  - `pathData` 合法性校验
  - `points` 合法性校验
  - 空文字 warning
  - 基于 `sourceSnapshot` 的安全写回保护

## 现有交互矩阵

### iframe 预览侧
- 负责检测 chart / SVG / 普通 HTML 元素。
- 对 SVG 负责命中目标元素、收集可编辑 item、构造 request，并通过 `postMessage` 发给父层。
- 同时还承担 SVG selection overlay、frame、preview patch、commit patch、session sync 等运行时逻辑。

### 父层 `HtmlPreview`
- 负责消费 iframe 消息。
- 在 SVG 场景下负责：
  - 接收编辑请求
  - 打开 SVG modal
  - 应用 patch 并写回 HTML
  - 与 context menu、patch error、其他编辑器状态互斥

### modal 编辑器
- 由 `SvgAssetEditorModal` 包装 `SvgTextCanvasEditor`。
- `SvgTextCanvasEditor` 既有画布，又有工具栏，又有右侧属性区，还自己维护 history、selection、drag state、connector binding 等复杂状态。

### HTML 源码写回
- SVG 修改最终不是直接改 DOM，而是转换成 SVG patch，再由 `applySvgPatch` 通过 locator 精确回写到 HTML 源码中。
- 写回前还会检查 tagName 和 `sourceSnapshot`，避免误写已经变化的源内容。

## 问题复盘
- 当前“编辑弹窗没有弹出”只是现象，根因是 SVG 编辑能力被拆散在多层且高度耦合。
- 现有链路没有形成和 chart 一样清晰的“检测 -> 打开 editor -> 提交 patch”模型，而是把选择、画布交互、overlay、消息同步、modal 和源码回写都塞进了一条长链里。
- 这使得任何一层的小偏差都可能表现为“入口失效”，而且真实环境与测试环境的命中表现不一致时尤其难排查。
- 因此本轮不再继续修现有实现，而是先完整下线，为后续重写腾出干净边界。

## 本轮范围

### 本轮要做
- 完整下线当前 SVG 编辑能力。
- 删除 SVG 编辑入口、消息、弹窗、patch、专用 helper、样式和测试。
- 保留 chart 编辑和通用 HTML visual-edit。
- 让 HTML preview 中的 inline SVG 回到普通预览内容，不再暴露坏掉的编辑入口。

### 本轮不做
- 不修现有 SVG 编辑链路。
- 不提供“重写中”占位弹窗或提示。
- 不实现新的 SVG 编辑器。
- 不调整 chart 编辑功能或 chart 数据模型。

## 重写目标
- 新的 SVG 编辑功能应采用清晰、单向的数据流：检测命中、打开编辑器、编辑本地草稿、一次性提交、统一写回。
- 新实现应尽量避免把复杂交互长期放在 iframe 宿主脚本内维护。
- SVG 编辑器应有明确的最小 v1 范围，优先保证“稳定打开、稳定修改、稳定写回”，而不是一次性恢复所有高阶能力。
- chart 编辑链路应继续独立存在，不与新的 SVG 编辑实现重新纠缠。

## 重写后的能力目标
- 稳定入口：
  - 任意 inline SVG 的编辑入口在桌面 WebView 中可稳定触发。
  - 入口策略单一明确，避免左键/双击/右键分别走不同隐式状态链。
- 稳定数据契约：
  - iframe 与父层之间若仍需桥接，只保留最小、稳定的 request 协议。
  - draft/selection/frame/history 不跨层分散持有。
- 稳定写回：
  - 仍保留基于 locator 的最小 patch 写回思路。
  - 保留基础并发保护，避免误写已变化源码。
- 可扩展：
  - 几何、样式、文字、路径能力分层清晰，便于后续逐项恢复。

## 非目标
- 不要求重写版一开始就支持当前所有 connector、bezier、batch edit、history、route binding 能力。
- 不要求重写版一开始就支持 SVG 的所有标签类型。
- 不要求重写版继续复用当前 `SvgTextCanvasEditor` 的内部状态结构。

## 约束与设计边界
- chart 编辑必须保持现状可用。
- 通用 HTML visual-edit 不应被 SVG 重写牵连。
- 重写时应优先保持代码边界清晰，而不是勉强复用现有大而全的 SVG 编辑器状态机。
- 若 SVG 需要再次进入 modal 模式，modal 本身应尽量薄，业务状态集中在新的 SVG 编辑域模型中。

## 失败处理要求
- 打不开编辑器时，要有可诊断的单一失败点，而不是静默丢消息。
- 无法安全写回时，要向用户给出明确失败反馈，不允许无声失败。
- 遇到不支持的 SVG 结构时，应明确降级，而不是部分可见、部分失效。

## 验收场景
- 下线阶段验收：
  - inline SVG 左键/双击/右键都不再打开编辑器。
  - context menu 不再出现 `Edit SVG`。
  - chart 点击与右键打开仍然正常。
  - HTML preview 其他模式不受影响。
- 未来重写版验收：
  - 至少一种稳定入口可以在桌面环境中 100% 打开 SVG 编辑器。
  - 基础图形与文字修改可以稳定提交并写回 HTML。
  - 不支持的结构有明确反馈。
  - chart 编辑不回归。

## 结论
- 当前 SVG 编辑实现已经不适合继续补丁式修复。
- 最优策略是先彻底下线旧链路，再以更小、更清晰、更可验证的边界完整重写。
- 本文档的作用是为后续 SVG 编辑重写提供统一的背景、范围、目标和验收基线。
